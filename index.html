文档 B：质化软规范 (灵魂)

《理论试题的设计及常见问题解析》深度拆解

核心理念：从“记忆”到“应用”

依据 Bloom 认知目标分类，试题必须完成以下转型：

记忆 (What)
A1型题(单句)应大幅减少。死记硬背不再是考核重点。
应用 (How to)
A2/A3/案例题是核心。考察分析判断与解决问题的能力。
A型题 (最佳选择)：
  • 叙述式病例：必须含完整要素(性别/年龄/主诉/检查)，拒接“假A2”。
  • 否定词加粗：如“不宜”、“不包括”，必须 加粗。
  • 避坑：题干冗长无效信息；语义模糊（常见/多见）。
B型题 (配伍)：
  • 同质性铁律：备选答案性质必须一致(如全抗生素)。
  • 引导句：必须有引导句说明关系；禁止使用否定引导句。
X型题 (多选)：
  • 难度定位：正确答案 ≥ 2个。
  • 绝对性：正确必须绝对正确，拒绝模棱两可。
案例分析题：
  • 选项特殊：每题 4个选项。
  • 信息控制：逐步释放信息，防泄题。
  • 严禁：“以上都是/不是”；选项间存在逻辑包含。
高质量试题“四字诀”
真 (真实情景) 准 (信息完整) 同 (选项同质) 难 (拒绝一眼假)

V3.0 程序设计架构深度解析

从“黑盒脚本”进化为“交互式 RAG 系统”。
本文将从数据流、API 策略、非结构化数据处理及 Prompt 架构四个维度，剖析 V3.0 的工程实现。

1. 整体架构：Client-Side RAG

V3.0 采用的是一种极简的 Serverless / Client-Side Architecture。它没有后端数据库，所有的数据处理（ETL）、逻辑控制（Logic）和渲染（View）都在浏览器端完成。

graph LR subgraph Browser_Client [浏览器客户端] UI[UI 配置层
(考点/题量)] --> Logic[业务逻辑层
(状态管理)] PDF[PDF 文件] --> Parser[PDF.js 解析器
(ETL)] Parser --> Context[全量文本上下文] Logic --> Prompt[Prompt 组装工厂] Context --> Prompt end subgraph LLM_Service [模型服务层] Prompt --> API_Call{API 网关
(Gemini 2.5/OpenAI)} API_Call --> Model[大语言模型] Model --> Result[Markdown 响应] end Result --> Renderer[Marked.js 渲染器] --> DOM[DOM 试卷]
  • 优势： 零部署成本，数据隐私性高（PDF 不上传服务器，直接在本地内存处理），响应速度仅受限于 LLM API。
  • 设计模式： 典型的 MVC (Model-View-Controller) 变体。State (SelectedKPs) 作为 Model，Prompt Injection 作为 Controller，HTML 作为 View。

2. 核心逻辑：动态注入与控制反转

V3.0 与 V1.0 最大的区别在于控制权的转移。V1.0 是硬编码（Hardcoding），V3.0 实现了动态注入（Dynamic Injection）。

状态管理 (State Management)

程序引入了 `selectedKPs` 数组来维护用户的意图。这是整个系统的“状态源”。

// 获取 UI 状态 const selectedKPs = Array.from( document.querySelectorAll('.kp-checkbox:checked') ).map(cb => cb.value); // 状态守卫 (Guard Clause) if (selectedKPs.length === 0) { alert("请至少选择一个考点"); return; }

提示词工厂 (Prompt Factory)

System Prompt 不再是静态字符串，而是根据状态动态构建的模板。

const systemPrompt = ` ... **【本次考试指定考点范围】**： 请**仅从**以下用户勾选的知识点范围内进行出题： ${selectedKPs.join('\n')} // ^^^ 动态注入点 (Injection Point) ... `;

3. API 引用策略：适配器模式

为了兼容 Gemini 和 OpenAI/DeepSeek，V3.0 实现了一个简单的适配器模式 (Adapter Pattern)。代码默认指向了最新的 Gemini 2.5 Flash 模型。

// 默认模型配置 (V3.0 Code Line 120+) <option value="gemini-2.5-flash-preview-09-2025">gemini-2.5-flash (最新)</option> // 策略分发 (Strategy Pattern) if (provider === 'gemini') { // 调用 Gemini API (v1beta) result = await callGemini(apiKey, model, system, user); } else { // 兼容所有 OpenAI 格式接口 result = await callOpenAI(apiKey, baseUrl, model, system, user); }

Context Window 处理： 由于 V3.0 采用全文投喂，API 调用时直接截取了前 70,000 字符。这是基于 Gemini 2.5 Flash (Preview) 及其前代 1.5 Pro 拥有百万级 Token 上下文窗口的特性设计的，避免了复杂的 RAG 切片，极大简化了系统。

4. PDF 解读：非结构化数据 ETL

系统如何理解 PDF？这里使用了一个“暴力提取” (Brute-force Extraction) 的策略。

处理流程：

  1. Stream Reading: 使用 `file.arrayBuffer()` 读取二进制流。
  2. Parsing: 调用 `pdfjsLib.getDocument()`。
  3. Iteration: 遍历每一页 (`pdf.numPages`)。
  4. Text Mining: 调用 `page.getTextContent()` 获取文本图元。
  5. Aggregation: 将所有文本拼接成一个巨大的字符串 `fullText`。

设计考量：

  • 为什么不保留格式？ LLM 对纯文本的理解能力极强，保留复杂的 PDF 布局（表格、分栏）反而会增加 Token 消耗且容易引入噪点。
  • 局限性： 图片和扫描件无法识别（因为没有集成 OCR），仅支持文本型 PDF。

总结：V3.0 的工程化思维

V3.0 不是简单的功能堆砌，而是体现了“以人为本”的工程设计。 它通过前端交互设计解决了 AI 输出不可控的问题， 通过升级到 Gemini 2.5 利用了更强的推理能力和上下文窗口， 是一个典型的“模型能力驱动应用架构简化”的案例。
